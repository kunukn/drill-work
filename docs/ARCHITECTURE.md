# Architecture

A high-level map for the architect.

For setup and commands, see [README.md](README.md). For conventions, see [.github/copilot-instructions.md](.github/copilot-instructions.md). For known UI-library quirks, see [BUGS.md](BUGS.md).

## In two sentences

Drill-work is a single-deployable freight-bookings app: TanStack Start (React 19 + Vite + Nitro) with a co-located server, Drizzle + `node:sqlite` for persistence, and Zod schemas as the contract that types both ends and produces an OpenAPI spec. UI routes and HTTP endpoints are file-routed in the same tree; "client" and "server" are folders, not deploy targets.

## Stack at a glance

| Layer                     | Choice                                                                                 |
| ------------------------- | -------------------------------------------------------------------------------------- |
| App framework             | TanStack Start (React 19 + Vite + Nitro), file-routed via TanStack Router              |
| Server runtime            | Nitro (co-located with the UI app)                                                     |
| DB                        | SQLite via `node:sqlite`, Drizzle ORM through `drizzle-orm/sqlite-proxy`               |
| Server state              | TanStack Query + `ky` HTTP client                                                      |
| Forms                     | `react-hook-form` + `zod` via `@hookform/resolvers`                                    |
| Validation / contracts    | Zod, with `@asteasolutions/zod-to-openapi`                                             |
| Styling                   | Tailwind v4 (Vite plugin)                                                              |
| UI library                | `@dfds-ui/navaigator` (pre-1.0; falls back to `src/components/generic/` then raw HTML) |
| Lint / format / typecheck | oxlint (Rust), Prettier, tsgo (TypeScript Native Preview)                              |

## System layers

```text
┌─────────────────────────────────────────────────────────────┐
│                       Browser                               │
│                                                             │
│  React components ─▶ TanStack Query hooks ─▶ ky requests    │
│  (src/components)    (src/lib/api.ts)        (HTTP/JSON)    │
└──────────────────────────────┬──────────────────────────────┘
                               │
┌──────────────────────────────▼──────────────────────────────┐
│  File-routed HTTP handlers   (src/routes/api/*.ts)          │
│  Thin adapters: parse → call server fn → json(...)          │
└──────────────────────────────┬──────────────────────────────┘
                               │
┌──────────────────────────────▼──────────────────────────────┐
│  Server resource modules    (src/server/api/*.ts)           │
│  Zod schemas + OpenAPI registration + data functions        │
└──────────────────────────────┬──────────────────────────────┘
                               │
┌──────────────────────────────▼──────────────────────────────┐
│  Drizzle ORM (sqlite-proxy)  →  node:sqlite  →  local.db    │
│  (src/server/db/*)                                          │
└─────────────────────────────────────────────────────────────┘
```

There is no separate API service. The same Vite/Nitro process serves both the React app and the JSON endpoints; only the folder tells you which side of the wire a piece of code runs on.

## Where code lives

| Path                                     | Purpose                                                                                                                                                                     |
| ---------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `src/routes/*.tsx`                       | UI routes (file-routed). `__root.tsx` is the shell + providers.                                                                                                             |

| `src/server/api/*.ts`                    | Resource modules — one file per entity (bookings, customers, sailings, terminals, vessels). Define schemas, register them with the OpenAPI registry, export data functions. |

| `src/server/db/index.ts`                 | DB client wiring (node:sqlite + sqlite-proxy adapter).                                                                                                                      |
| `src/server/openapi.ts`                  | Registry + `extendZodWithOpenApi`. Must not import `./api` (circular).                                                                                                      |
| `src/server/openapi-spec.ts`             | Imports `./api` to trigger registrations, then generates the spec.                                                                                                          |
| `src/server/latency.ts`                  | 200–300 ms artificial delay so optimistic-UI code is exercised in dev.                                                                                                      |

| `src/lib/booking-form-schema.ts`         | Plain-Zod form schema, mirrored from server schemas to avoid pulling drizzle into the client bundle.                                                                        |

| `src/components/generic/`                | Cross-feature primitives (e.g. `StatusBadge`).                                                                                                                              |

## Data flow — a single CRUD trace

Creating a booking, end to end:

1. **UI** — `BookingNew` calls `useCreateBooking()` ([src/lib/api.ts:184](src/lib/api.ts#L184)) and `mutateAsync(values)`.
2. **Wrapper** — `useCreateBooking` invokes `request<BookingT>("/api/bookings", { method: "POST", body: ... })`. `request()` ([src/lib/api.ts:57](src/lib/api.ts#L57)) wraps `ky` to normalize errors into `ApiError` and short-circuit 204s.
3. **Transport** — `ky` POSTs to the same origin.
4. **HTTP handler** — `src/routes/api/bookings.ts` POST handler awaits `simulateLatency()`, parses JSON, calls `createBooking(body)`, and returns `json(created, { status: 201 })`. Any `z.ZodError` becomes a 400 with `{ error }`.
5. **Resource module** — `createBooking()` in `src/server/api/bookings.ts` parses the body through `NewBookingInput`, generates an id, inserts via Drizzle, and re-fetches the row (sqlite-proxy doesn't surface affected-row counts, so a follow-up read is the safest way to return the canonical record).
6. **DB** — Drizzle compiles the SQL; `sqlite-proxy` ([src/server/db/index.ts](src/server/db/index.ts)) maps named-column results into the positional row arrays Drizzle expects.
7. **Response** — `request()` resolves with the typed `BookingT`. `useCreateBooking.onSuccess` calls `qc.invalidateQueries({ queryKey: ["bookings"] })`, marking every cached `["bookings", <filter>]` variant stale. Any mounted list view refetches.

## OpenAPI is a derived artifact

Zod schemas are declared with `.openapi(...)` annotations in each resource module and registered (`registry.register(...)`, `registry.registerPath(...)`) at module load — these are **side effects**. `src/server/api/index.ts` imports every resource so the registrations run; `src/server/openapi-spec.ts` imports `./api` and calls `OpenApiGeneratorV31` to produce the spec at runtime. Two non-obvious rules:

- `src/server/openapi.ts` must **not** import from `./api`. Adding the import creates a circular dep that re-orders evaluation and silently disables `.openapi()` on Zod schemas.
- Every new resource needs a side-effect import in `src/server/api/index.ts`, or its paths/schemas vanish from the spec.

The generated spec is served at `/api/openapi.json`; the docs UI at `/api/docs`.

## Frontend patterns (named)

The app is small, so the patterns are concentrated in a few files. This list maps what you'll see in the code to vocabulary you already have.

### Factory pattern — `queryKeys`

[src/lib/api.ts:85](src/lib/api.ts#L85). A single object whose methods build cache keys (`bookings(filter)`, `booking(id)`, `customers()`, …). Mutations invalidate by prefix (`["bookings"]` matches every `["bookings", <filter>]` variant), so funneling key construction through the factory keeps reads and invalidations in lockstep. Add a new resource here before using it in a hook.

### Wrapper / Adapter — `request<T>()`

[src/lib/api.ts:57](src/lib/api.ts#L57). Wraps `ky` with two responsibilities: (a) parse the response, treating 204 No Content as `undefined`; (b) translate `ky`'s `HTTPError` into a domain-specific `ApiError` carrying the server's `{ error }` string. Every hook in `src/lib/api.ts` goes through this wrapper, so error handling is uniform at every call site.

### Custom hooks as Facade

`useBookings`, `useBooking`, `useCreateBooking`, `useUpdateBooking`, `useDeleteBooking`, `useCustomers`, `useVessels`, `useSailings`, `useTerminals`. Each hides TanStack Query mechanics (key, query/mutation fn, lifecycle hooks, cache surgery) behind a domain verb. Components don't import `useQuery` or `useMutation` directly.

### Provider pattern

`QueryClientProvider` mounted at the root in [src/routes/\_\_root.tsx](src/routes/__root.tsx). The client itself comes from `getQueryClient()` ([src/lib/queryClient.ts](src/lib/queryClient.ts)) — a per-request instance on SSR (no cache leakage between users) and a module-level singleton in the browser (cache survives navigation).

### Controller pattern (form integration)

react-hook-form supports two integration modes: `register("field")` for native HTML inputs (uncontrolled, ref-based), and `<Controller>` for controlled non-input components. `BookingFields` ([src/components/bookings/BookingFields.tsx](src/components/bookings/BookingFields.tsx)) uses `Controller` for NavAIgator's `Select` (which is a Radix-style controlled component) and `register` for the rest. Numeric `<input type="number">` uses `register(name, { valueAsNumber: true })` so RHF coerces the string to a number.

### Schema-first validation (single source of truth at the edges)

Zod schemas in `src/server/api/*` define the request/response shape, the inferred TypeScript types (`z.infer<typeof Booking>`), the runtime validation, _and_ the OpenAPI annotations. Form schemas in `src/lib/` are plain-Zod **mirrors** of the relevant fields — a deliberate copy, because importing the server schemas into client code would pull in `drizzle-orm`. The duplication is a contract boundary, not technical debt.

### Side-effect registration

Resource modules register OpenAPI paths and schemas as a side effect of being imported. This pattern is invisible in the call graph (no one calls `register(...)` from outside the module) — and that's exactly what makes the rule "import the new resource in `src/server/api/index.ts`" so easy to forget. Treat the index file as the registry's manifest.

### Optimistic update with snapshot/rollback

`useDeleteBooking` ([src/lib/api.ts:255](src/lib/api.ts#L255)) is the canonical example. `onMutate` cancels in-flight refetches, snapshots the current cache for every `["bookings", <filter>]` variant, and removes the row optimistically. `onError` restores from the snapshot. `onSettled` invalidates so the server is the final source of truth. `useUpdateBooking` and `useCreateBooking` are non-optimistic by deliberate choice — see comments above each hook.

### Mapped errors at boundaries

- `useBooking` ([src/lib/api.ts:122](src/lib/api.ts#L122)) maps a 404 to a resolved `null` instead of an error. Callers branch on `data` (loading: `undefined`, missing: `null`, found: `BookingT`) without a separate error-state branch.
- HTTP handlers catch `z.ZodError` and respond with 400 + `{ error }`.

The pattern: where two error vocabularies meet, translate at the boundary so the inner code only sees one.

### Sentinel value

`const ALL = "__all__"` in [src/components/bookings/BookingsList.tsx](src/components/bookings/BookingsList.tsx). Radix `<Select>` rejects `""` as an item value, so the "no filter" option uses a sentinel that's translated to `undefined` in URL state. See BUGS.md #4.

### Stable identity / module-level constants

`EMPTY_BOOKINGS` in `Dashboard.tsx`. Module-level `[]` to avoid re-allocating an empty array per render. Cheap, harmless, occasionally useful when paired with referentially-aware children.

### Auto-imports as ambient globals

Common identifiers (most React hooks, NavAIgator primitives, route helpers, app-local symbols like `StatusBadge` and `STATUS_META`) are injected as ambient globals via `unplugin-auto-import` (see `auto-import.config.ts`). **Caveat**: TanStack Start emits a `?tsr-split=component` chunk per route component that the auto-import transform doesn't reach — so files in `src/routes/` need explicit imports for everything they use. Auto-imports are intended for `src/components/`, `src/hooks/`, `src/lib/`.

## State boundaries

| State                              | Owner                | Lives in                              |
| ---------------------------------- | -------------------- | ------------------------------------- |
| Server data (bookings, vessels, …) | TanStack Query cache | `useQuery` hooks via `src/lib/api.ts` |
| Form state                         | react-hook-form      | `useForm` per form                    |
| URL state (filters, route params)  | TanStack Router      | `validateSearch` (Zod) per route      |
| Local UI state (dialogs, etc.)     | `useState`           | Co-located with the component         |

There is no Redux/Zustand/Jotai. None is needed at this size; if it grows, the cache + URL split scales further before a global store is justified.

## Build pipeline

`vite.config.ts` plugin order: `viteTsConfigPaths` → `tailwindcss` (v4 Vite plugin) → `tanstackStart` → `nitro` (with `node:*` externals) → `viteReact`. Path alias `@/* → ./src/*` is mirrored in `tsconfig.json`. `pnpm build` runs `setup.ts` + `pnpm typecheck` first, then `vite build`. Output is a Nitro server at `.output/server/index.mjs`.

## What this architecture optimizes for

- **Single deployable.** UI and API ship together; no cross-service contract drift.
- **Schema-first contracts.** One Zod definition produces TS types, runtime validation, and the OpenAPI spec.
- **Fast feedback loop.** oxlint + tsgo + Prettier are sub-second in practice; pre-push checks parallelize typecheck and lint. Shift-left: only clean code reaches the remote.
- **Low surface area.** No global store, no GraphQL, no service mesh. The smaller the architecture, the less there is to misuse.

Trade-offs worth naming:

- **SQLite + single Nitro process.** Vertical scale only. Fine for this workload; would migrate to Postgres + a separate API process if write volume or concurrency demanded it.
- **Co-located UI/API.** Fast dev loop; harder to scale teams independently. Re-evaluate at ~10+ engineers.
- **NavAIgator UI library is pre-1.0.** Several documented quirks (BUGS.md). Wrapping it in a thin generic layer (`src/components/generic/`) is encouraged so the blast radius of an upgrade is contained.
